# LocalAreaInterconnection 功能设计方案

## 1. 产品定位

LocalAreaInterconnection 是一款面向 Steam/PC 局域网联机游戏的虚拟局域网工具。

目标是让分处两地的玩家在互联网上组成一个虚拟 LAN，使只支持“局域网联机”的游戏可以发现房间或通过虚拟 IP 直连。

一句话定位：

> 为异地玩家创建一个低门槛、低延迟、可诊断的虚拟局域网联机环境。

## 2. 核心问题

很多 PC 游戏只提供 LAN 模式，玩家不在同一个物理局域网时会遇到：

- 游戏房间列表无法发现对方。
- 游戏只广播局域网搜索包，不提供公网房间。
- 只能通过局域网 IP 加入。
- 普通 VPN 能 ping 通，但游戏房间仍然不出现。
- 防火墙、网卡优先级、端口、NAT 类型导致联机失败。

本产品要解决的不是 Steam 账号、游戏授权或反作弊问题，而是提供一个让游戏进程可见的虚拟局域网。

## 3. 设计原则

- 优先 P2P：游戏流量尽量玩家之间直连，不经过服务器。
- 无业务后端优先：不做账号系统、不保存用户游戏数据、不托管房间历史。
- 小白可用：创建房间、复制邀请码、加入房间、启动游戏。
- 可诊断：联机失败时明确告诉用户卡在哪里。
- 不承诺万能：不同游戏的 LAN 实现不同，需要逐步适配。
- 安全默认：房间隔离、端到端加密、最小权限。
- 先做玩家电脑互联，不先做整个家庭局域网互联。

## 4. 目标用户

### 4.1 核心用户

- 和朋友异地玩老游戏、独立游戏、局域网合作游戏的玩家。
- 习惯在 Steam 中玩游戏，但游戏本身只支持 LAN 联机。
- 不懂路由器端口转发、不想折腾传统 VPN 的普通玩家。

### 4.2 次级用户

- 小型电竞/联机社群。
- 怀旧游戏玩家。
- 开发者测试 LAN 联机功能。
- 临时组局的朋友群。

## 5. 非目标

第一阶段不做：

- 绕过正版验证、DRM、反作弊或平台限制。
- 游戏加速器式全局代理。
- 云游戏串流。
- 大规模公共匹配大厅。
- 企业内网互联。
- 手机端作为主要联机节点。
- 完整二层跨家庭局域网桥接。

## 6. 核心概念

### 6.1 房间

房间是一次虚拟局域网会话。每个房间有：

- 房间 ID。
- 房主。
- 成员列表。
- 房间密钥。
- 虚拟网段。
- 连接策略。
- 可选游戏配置。

### 6.2 虚拟网卡

每个玩家电脑加入房间后，客户端创建或启用一个虚拟网卡，并分配虚拟 IP。

示例：

```text
玩家 A: 10.77.0.2
玩家 B: 10.77.0.3
玩家 C: 10.77.0.4
```

游戏通过这个虚拟网卡收发局域网流量。

### 6.3 广播转发

很多 LAN 游戏通过广播发现房间，例如：

```text
255.255.255.255
10.77.0.255
UDP broadcast
Multicast/mDNS/SSDP-like discovery
```

客户端需要捕获虚拟网卡上的广播/组播包，并转发给同房间成员。

### 6.4 直连与中继

优先使用 UDP P2P 直连。

如果 NAT 环境不允许直连，可选择：

- 提示用户切换网络。
- 提示房主端口转发。
- 使用可选中继服务。
- 降级到手动直连模式。

## 7. 产品模式

### 7.1 零后端手动模式

适合技术用户或早期 MVP。

特征：

- 不需要账号。
- 不需要中心服务器。
- 房主提供公网 IP、端口或 DDNS。
- 朋友输入连接信息加入。
- 游戏流量 P2P。

限制：

- 需要房主网络可入站访问。
- NAT 复杂时失败率高。
- 不适合普通用户作为默认模式。

### 7.2 轻量协调模式

推荐作为正式产品默认模式。

协调服务只做：

- 创建临时房间。
- 交换节点公网地址和候选地址。
- 协助 NAT 打洞。
- 传递一次性密钥材料。
- 短期保活。

协调服务不做：

- 不转发游戏流量。
- 不保存聊天记录。
- 不保存游戏数据。
- 不做好友关系。
- 不做长期账号体系。

这是“无业务后端”的折中方案：需要一个很轻的联网协调点，但不承载核心业务数据。

### 7.3 中继兜底模式

用于 P2P 失败时。

特点：

- 延迟更高。
- 带宽成本明显增加。
- 可作为付费能力或实验能力。
- 第一阶段可以不做，只显示“当前网络无法直连”。

## 8. MVP 范围

### 8.1 平台

第一阶段只做 Windows。

原因：

- Steam PC 游戏主要在 Windows。
- 玩家群体集中。
- 虚拟网卡、Windows 防火墙、网卡优先级是核心问题。
- 先控制复杂度。

后续再考虑：

- Linux。
- Steam Deck。
- macOS。

### 8.2 MVP 功能清单

#### 房间功能

- 创建房间。
- 加入房间。
- 复制邀请码。
- 显示房间成员。
- 显示每个成员虚拟 IP。
- 退出房间。
- 房主关闭房间。

#### 虚拟网络

- 自动安装/检测虚拟网卡驱动。
- 分配虚拟 IP。
- 建立玩家之间的加密隧道。
- 支持 IPv4。
- 支持 UDP 单播。
- 支持 UDP 广播转发。
- 支持基础 TCP 转发。

#### 游戏辅助

- 显示“推荐在游戏中选择的 IP”。
- 一键复制自己的虚拟 IP。
- 显示当前游戏可能需要的连接方式：
  - 自动发现。
  - Direct IP。
  - 主机建房后其他人加入。
- 允许用户手动添加游戏端口规则。

#### 诊断

- P2P 是否成功。
- 延迟。
- 丢包。
- NAT 类型粗略判断。
- 防火墙是否阻止。
- 虚拟网卡是否启用。
- 两名玩家是否能互 ping。
- 广播包是否被转发。
- 游戏端口是否有流量。

#### 安全

- 房间邀请码带随机密钥。
- 房间内流量加密。
- 房间成员隔离。
- 退出房间后关闭路由和隧道。
- 默认不暴露真实局域网。

### 8.3 MVP 不做

- 公共游戏大厅。
- 好友系统。
- 聊天系统。
- 语音系统。
- 全局流量代理。
- 整个家庭 LAN 桥接。
- 自动识别所有 Steam 游戏。
- 移动端控制器。
- 游戏加速线路。

## 9. 用户流程

### 9.1 房主创建房间

```text
打开客户端
  -> 点击“创建房间”
  -> 选择游戏或通用模式
  -> 客户端创建虚拟网段
  -> 客户端启动隧道监听
  -> 生成邀请码
  -> 复制邀请码发给朋友
```

界面显示：

- 房间名称。
- 邀请码。
- 我的虚拟 IP。
- 当前连接状态。
- 推荐操作：“启动游戏并创建 LAN 房间”。

### 9.2 朋友加入房间

```text
打开客户端
  -> 点击“加入房间”
  -> 粘贴邀请码
  -> 客户端解析房间信息
  -> 创建虚拟网卡
  -> 建立 P2P 隧道
  -> 获得虚拟 IP
  -> 测试与房主连通性
```

界面显示：

- 已加入房间。
- 房主虚拟 IP。
- 当前延迟。
- 广播发现状态。
- 推荐操作：“进入游戏 LAN 页面查找房间”。

### 9.3 游戏无法发现房间

客户端进入诊断流程：

```text
检查虚拟网卡
  -> 检查双方 P2P 连通
  -> 检查 UDP 广播
  -> 检查 Windows 防火墙
  -> 检查游戏是否绑定错误网卡
  -> 建议尝试 Direct IP
```

给用户的错误提示要具体：

- “你们可以互 ping，但没有检测到游戏广播包。”
- “Windows 防火墙可能阻止了该游戏的入站 UDP。”
- “游戏可能选择了物理网卡而不是虚拟网卡。”
- “当前网络无法 P2P 直连，需要房主端口转发或使用中继。”

## 10. 网络设计

### 10.1 虚拟网段

默认使用私有地址段：

```text
10.77.0.0/16
```

每个房间使用一个子网：

```text
10.77.12.0/24
```

避免与用户本地网络冲突。

如果检测到冲突，自动换段：

```text
10.88.0.0/16
172.22.0.0/16
```

### 10.2 地址分配

MVP 可使用房主分配：

```text
房主: 10.77.x.1
成员 1: 10.77.x.2
成员 2: 10.77.x.3
```

后续可改为确定性分配：

```text
IP = room_subnet + hash(peer_id)
```

### 10.3 数据包路径

```text
游戏进程
  -> 虚拟网卡
  -> 客户端捕获 IP/UDP/TCP 包
  -> 加密封装
  -> UDP P2P 隧道
  -> 对方客户端解密
  -> 对方虚拟网卡
  -> 对方游戏进程
```

### 10.4 广播处理

广播包不应无限制泛洪。

策略：

- 只转发房间虚拟网段内广播。
- 默认转发 UDP 广播。
- 限速广播包。
- 可按游戏端口白名单转发。
- 不默认转发物理局域网广播。

### 10.5 协议选择

可选路线：

#### 路线 A：自研轻量隧道

优点：

- 产品体验可控。
- 可针对游戏广播优化。
- 不依赖第三方组网平台。

缺点：

- 虚拟网卡、NAT、加密、稳定性都要自己处理。
- 开发周期更长。

#### 路线 B：封装 ZeroTier/SoftEther

优点：

- 快速验证需求。
- 网络底层成熟。
- 可先做 UI、诊断、游戏适配。

缺点：

- 依赖第三方。
- 商业化和分发需要确认协议。
- 产品差异化较弱。

#### 路线 C：先用 WireGuard 思路做三层，补广播代理

优点：

- 安全模型成熟。
- 性能好。
- 实现难度低于完整二层桥接。

缺点：

- 不是完整二层 LAN。
- 部分只依赖广播发现的游戏需要额外适配。

建议：

第一阶段可先验证路线 B 或 C；如果需求成立，再投入路线 A。

## 11. 客户端模块

### 11.1 UI 模块

页面：

- 首页。
- 创建房间。
- 加入房间。
- 房间详情。
- 连接诊断。
- 游戏配置。
- 设置。
- 日志导出。

### 11.2 房间模块

职责：

- 生成房间 ID。
- 生成邀请码。
- 解析邀请码。
- 管理成员状态。
- 管理房间密钥。
- 维护房间生命周期。

### 11.3 虚拟网卡模块

职责：

- 安装/检测驱动。
- 创建虚拟网卡。
- 设置 IP、掩码、MTU。
- 管理网卡启停。
- 设置路由。
- 检测网卡优先级。

### 11.4 隧道模块

职责：

- 建立 P2P 连接。
- 加密/解密数据包。
- 封装/解封装虚拟网络包。
- 心跳保活。
- 重连。
- 统计延迟、丢包、吞吐。

### 11.5 广播代理模块

职责：

- 捕获虚拟网卡广播包。
- 按规则转发给房间成员。
- 控制广播风暴。
- 记录广播诊断信息。

### 11.6 NAT 穿透模块

职责：

- 检测本地候选地址。
- UDP 打洞。
- 判断直连是否成功。
- 选择连接路径。
- 输出诊断结论。

### 11.7 防火墙模块

职责：

- 检测 Windows 防火墙状态。
- 提示添加入站规则。
- 为客户端添加必要规则。
- 识别游戏进程是否被阻止。

### 11.8 诊断模块

职责：

- 连通性测试。
- 虚拟 IP ping 测试。
- UDP 收发测试。
- 广播发现测试。
- 端口流量检测。
- 日志打包。

## 12. 邀请码设计

邀请码可包含：

```json
{
  "version": 1,
  "room_id": "abc123",
  "room_name": "Friday LAN",
  "mode": "p2p",
  "virtual_subnet": "10.77.12.0/24",
  "host_peer_id": "peer_xxx",
  "host_endpoint": "optional-public-endpoint",
  "coordination_endpoint": "optional",
  "join_token": "random-secret"
}
```

用户看到的可以是短码或长文本。

MVP 可先使用长邀请码，后续再做短码。

## 13. 游戏适配

### 13.1 通用模式

默认适配：

- UDP 广播发现。
- Direct IP 连接。
- 常规 TCP/UDP 局域网通信。

### 13.2 游戏配置模板

后续可维护本地模板库：

```json
{
  "game_name": "Example Game",
  "steam_app_id": "123456",
  "discovery": "udp_broadcast",
  "ports": [27015, 27016],
  "join_method": "lan_list_or_direct_ip",
  "notes": "需要允许游戏通过 Windows 防火墙"
}
```

模板可以随客户端发布，不一定需要后端动态下发。

### 13.3 适配等级

- A 级：自动发现房间，体验接近真实局域网。
- B 级：不能自动发现，但 Direct IP 可用。
- C 级：需要手动端口或特殊设置。
- D 级：不支持或受游戏/平台限制。

## 14. 安全设计

### 14.1 默认隔离

默认只允许房间内玩家访问虚拟 IP，不允许访问玩家真实局域网。

不要默认开启：

- 访问家庭 NAS。
- 访问路由器。
- 访问打印机。
- 转发物理 LAN 广播。

### 14.2 加密

要求：

- 房间密钥随机生成。
- 节点身份密钥本地生成。
- 数据包端到端加密。
- 邀请码泄露后可重新生成房间。

### 14.3 权限控制

MVP：

- 房主可踢人。
- 房主可关闭房间。
- 加入需要邀请码。

后续：

- 房间密码。
- 成员批准。
- 临时封禁。
- 只允许好友加入。

### 14.4 风险提示

用户加入房间后，相当于和其他玩家处于同一虚拟网络。客户端应提示：

- 不要加入陌生房间。
- 不要在房间内运行不可信服务。
- 不要把真实局域网共享给陌生人。

## 15. 性能目标

MVP 目标：

- 两人 P2P 成功时，额外延迟低于 5ms。
- CPU 占用低于 5%。
- 普通 LAN 游戏稳定运行 2 小时。
- 支持 2-8 人房间。
- 支持基础广播转发限速。

非目标：

- 不保证跨国低延迟。
- 不保证竞技游戏加速效果。
- 不承诺替代专业加速器。

## 16. 兼容性风险

### 16.1 游戏绑定网卡

部分游戏会选择错误网卡，导致房间不可见。

应提供：

- 网卡优先级建议。
- 虚拟网卡状态检测。
- Direct IP 兜底。

### 16.2 Windows 防火墙

首次运行游戏时 Windows 可能阻止入站连接。

应提供：

- 防火墙规则检测。
- 引导用户允许专用网络。
- 可选自动添加规则。

### 16.3 NAT 限制

校园网、公司网、部分运营商 CGNAT 可能导致 P2P 失败。

应提供：

- 直连失败原因提示。
- 手动端口转发指南。
- 可选中继方案。

### 16.4 反作弊与平台限制

部分游戏可能阻止虚拟网卡、VPN 或非官方联机方式。

应明确说明：

- 本产品只支持游戏本身允许的局域网联机。
- 不绕过反作弊、封禁、正版验证或平台规则。

## 17. UI 设计草案

### 17.1 首页

主要操作：

- 创建房间。
- 加入房间。

辅助信息：

- 最近房间。
- 网络状态。
- 虚拟网卡状态。

### 17.2 房间页

显示：

- 房间名称。
- 邀请码。
- 我的虚拟 IP。
- 成员列表。
- 每个成员延迟。
- 连接路径：直连/中继/失败。
- 广播状态。

操作：

- 复制邀请码。
- 复制我的 IP。
- 打开诊断。
- 退出房间。

### 17.3 诊断页

诊断项：

- 虚拟网卡：正常/异常。
- 隧道连接：正常/异常。
- P2P 直连：成功/失败。
- 防火墙：允许/可能阻止。
- 广播转发：有流量/无流量。
- 游戏流量：检测到/未检测到。

每个异常都要给下一步动作。

## 18. 技术路线建议

### 18.1 快速验证路线

目标：先验证“玩家愿不愿意用”。

方案：

- 客户端 UI：Tauri 或原生 Windows UI。
- 底层网络：先封装 ZeroTier 或 SoftEther。
- 增强能力：做游戏诊断、模板、操作引导。

优点：

- 快速出 Demo。
- 重点验证交互和需求。

缺点：

- 底层不完全自主。
- 差异化有限。

### 18.2 自主产品路线

目标：做真正独立的游戏虚拟 LAN 工具。

方案：

- 客户端 UI：Tauri。
- 核心服务：Rust。
- Windows 虚拟网卡：TAP/Wintun 类方案，需根据二层/三层目标选择。
- 加密隧道：Noise/WireGuard 思路。
- NAT 穿透：UDP hole punching。
- 广播代理：自研。

优点：

- 产品可控。
- 可针对游戏 LAN 做深度优化。

缺点：

- 开发难度明显更高。
- 驱动、签名、安装、权限、稳定性是长期成本。

### 18.3 推荐推进方式

阶段 1：做产品原型和网络验证。

阶段 2：做 Windows 客户端 MVP。

阶段 3：选 5-10 款游戏做适配测试。

阶段 4：根据失败案例完善诊断。

阶段 5：再决定是否自研完整隧道。

## 19. 里程碑

### M0：调研验证

目标：

- 确认 10 款目标游戏。
- 确认它们的 LAN 联机机制。
- 对比 Radmin VPN、ZeroTier、SoftEther。
- 写出第一版技术决策。

产出：

- 游戏兼容性表。
- 技术选型说明。
- MVP 范围确认。

### M1：本机虚拟网络实验

目标：

- 在两台 Windows 机器之间建立虚拟 IP 互通。
- 能 ping 通。
- 能发送 UDP。
- 能捕获广播包。

产出：

- 网络 PoC。
- 日志。
- 限制说明。

### M2：房间原型

目标：

- 创建房间。
- 加入房间。
- 分配虚拟 IP。
- 显示成员在线。

产出：

- 可运行 Demo。
- 基础 UI。

### M3：游戏联机 MVP

目标：

- 支持至少 2 款 LAN 游戏成功联机。

- 支持防火墙诊断。

## 20. 成功指标

### 技术指标

- 双人房间创建成功率。
- P2P 连接成功率。
- 平均额外延迟。
- 30 分钟内断线次数。
- 广播发现成功率。

### 产品指标

- 用户能否在 3 分钟内完成第一次联机。
- 联机失败时是否知道下一步该做什么。
- 是否愿意推荐给朋友。
- 是否愿意为稳定中继或高级诊断付费。

## 21. 后续可能的商业化

可选方向：

- 免费基础 P2P。
- 付费中继流量。
- 高级游戏适配模板。
- 多人房间增强。
- 房间历史和好友列表。
- 社群房间。
- Steam 游戏自动识别。

注意：

- 不要一开始就做复杂账号系统。
- 不要过早做公共大厅。
- 先验证“能不能稳定帮玩家连上”。

## 22. 关键风险

- P2P 在复杂 NAT 下失败率较高。
- 不同游戏 LAN 行为差异很大。
- 虚拟网卡驱动安装可能吓退用户。
- Windows 防火墙会制造大量误判。
- 没有中继时用户体验可能不稳定。
- 自研网络栈的长期维护成本高。
- 商业化时需要处理带宽成本。

## 23. 当前建议结论

这个方向可以做，且比普通无后端工具更有差异化。

建议第一版不要追求完整“真实二层局域网”，而是实现：

- 玩家电脑之间的虚拟 LAN。
- P2P 加密隧道。
- UDP 广播转发。
- Direct IP 兜底。
- 清晰的联机诊断。

产品成败的关键不只是网络能不能通，而是失败时能不能告诉普通玩家“为什么不行、下一步怎么做”。
